The problem it was solving

Outbound prospecting in CRE advisory is almost entirely manual and inconsistent. Research quality depends on who's doing it and how much time they have. Personalizing cold outreach for 40 or 100 contacts is practically impossible at that scale without either cutting corners on quality or spending days on prep work.

Most teams default to generic sequences or semi-personalized templates that feel thin. The compounding problem: even when there's a strong contact list, the friction between "we have this list" and "sequences are live and LinkedIn is running" is significant enough that campaigns get delayed, partially executed, or abandoned.

The key insight: the highest-friction points in outbound prospecting aren't sending emails — they're the research, the personalization, and the CRM setup that has to happen before anything goes out. That's where the system targets.

The honest evaluation

Brutal honesty section

The core pipeline works and is producing real outputs. Sequences are live, contacts are enrolled, LinkedIn is running. That's not a prototype — that's an operational system.

The qualifications are real though. The pipeline tracker is partially out of sync with HubSpot — enrollment status, LinkedIn connection status, and some field states aren't written back automatically. It requires periodic manual reconciliation.

The LinkedIn bot has to run locally, which means it can't be fully scheduled or run unattended. Research and sequence generation still require batched sessions — not fully continuous.

There was a field mapping bug — LinkedIn content pushed into email_body_3 — that required manual diagnosis and a corrective HubSpot push. The pipeline doesn't have a validation layer that catches that class of error before it gets to HubSpot. It should.

Decisions worth explaining

Why Selenium instead of Claude for LinkedIn

Claude's computer use capability can handle browser automation, but it burns API credits on every action and is slower. A local Selenium script with randomized delays sends connection requests in the background for zero Claude API cost. The results log back to the tracker automatically. For this specific task — clicking through a predictable UI — Selenium is the right tool.

The preflight check

Before any contact gets enrolled, the system checks three things: are they already in an active sequence, have we contacted them in the last 60 days, and did a previous email bounce? This kind of hygiene is usually an afterthought in manual campaigns. It causes real relationship damage when it breaks — the system makes it structural rather than optional.

A/B track routing

Not every contact gets a fully personalized sequence. If the AI research comes back with low confidence — no news triggers, thin LinkedIn presence — the contact gets routed to a market-segment track instead. Honest about what the system can and can't do, rather than generating low-quality personalization and calling it done.

The real point

Most people at an early stage in a brokerage or advisory role are learning the tools their firm already uses. They're not building the tools.

What's unusual here isn't that the system is finished — it isn't. It's that the problem decomposition is real and the execution is grounded. The preflight check logic, the field mapping architecture, the decision to use Selenium instead of burning Claude credits — these reflect genuine operational thinking, not a surface-level AI experiment.

Building something that surfaces a bug, diagnoses the root cause (column index off-by-one in a tracker push), and fixes it across three live HubSpot contacts in a single session is a different category of work than building a demo.

What I'd rebuild first

  • Validation layer before HubSpot push — the field mapping bug would have been caught instantly with a schema check before any data moves. This is the most important missing piece.
  • Tracker sync — the pipeline-master.csv being out of sync with HubSpot means I can't trust the tracker as a source of truth. A proper write-back step after each enrollment would fix this.
  • Move LinkedIn off local execution — the Selenium script being tied to a local browser is the biggest operational constraint. Worth exploring a hosted option once the core pipeline is stable.
